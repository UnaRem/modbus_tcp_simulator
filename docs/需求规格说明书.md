# 项目名称：Linux 高级 Modbus TCP 设备仿真系统

**版本：** 1.0

**适用平台：** Linux (CentOS/Ubuntu/Debian)、Windows

---

- ## 1. 引言 (Introduction)

  ### 1.1 项目背景

  在 EMS (能量管理系统) 开发中，开发者常面临真实的储能变流器 (PCS) 和电池管理系统 (BMS) 缺失的问题。现场环境通常是通过串口服务器或 `mbusd` 将 RS485 总线上的多个设备（PCS, BMS, 电表等）映射到一个 TCP 端口。现有模拟器无法模拟这种 **“单端口多设备”** 的拓扑，也无法模拟设备间的**物理能量流动**（如 PCS 充电导致 BMS 电量上升）。

  ### 1.2 核心目标

  1. **总线拓扑仿真**：支持在单个 TCP 端口上模拟多个 Modbus Slave ID（模拟 RS485 链路）。
  2. **物理闭环联动**：支持跨设备的逻辑控制与物理仿真（如功率积分计算电量）。
  3. **智能校验与自控**：支持数值范围限制、自动复位、心跳自增等逻辑。

  ------

  ## 2. 系统架构 (System Architecture)

  系统采用 **虚拟总线 (Virtual Bus)** 架构设计。

  ### 2.1 核心组件更新

  1. **Virtual Bus Controller (虚拟总线控制器)**:
     - 取代简单的 1对1 映射。
     - 该组件监听一个 TCP 端口（如 502），内部维护一个 `SlaveID -> DataBlock` 的路由表。
     - 当收到 TCP 请求时，根据协议中的 `Unit ID` 字段，将请求分发给对应的虚拟设备（PCS 或 BMS）。
  2. **Global Physics Engine (全局物理引擎)**:
     - 拥有上帝视角，可以同时读取 PCS 的寄存器（功率）和 BMS 的寄存器（SoC）。
     - 执行 $E = \int P dt$ 等跨设备物理运算。

  ------

  ## 3. 功能需求 (Functional Requirements)

  ### FR-01: 拓扑与并发 (Topology & Concurrency)

  - **FR-01-1**: **多从站单端口模式 (Multi-Slave Single Port)**: 系统必须支持在一个 TCP 端口上挂载多个虚拟设备，每个设备由唯一的 Slave ID 区分。此功能用于完美模拟硬件串口服务器或 `mbusd` 的行为。
  - **FR-01-2**: **多端口并发**: 支持同时开启多个 TCP 监听端口，每个端口下均可配置独立的 Slave ID 列表。

  ### FR-02: 写入值约束 (Value Constraints)

  - **FR-02-1**: **范围限制**: 支持定义寄存器的 `min/max` 值。例如 PCS 功率设定不能超过额定功率，否则返回 Modbus Exception 03。

  ### FR-03: 跨设备逻辑联动 (Cross-Device Logic) - *新增核心需求*

  - **FR-03-1**: **全局寻址**: 逻辑脚本必须支持 `Device(Name).Register(Addr)` 的寻址方式，允许脚本跨设备读写。
  - **FR-03-2**: **能量积分模型**: 内置或支持编写“功率-电量”积分逻辑。
    - 输入：PCS 功率寄存器 (kW)。
    - 状态：电池当前容量 (kWh)。
    - 输出：BMS SoC 寄存器 (%)。
    - 逻辑：`SoC_new = SoC_old + (Power * Time_Delta / Capacity)`.

  ### FR-04: 硬件行为仿真 (Hardware Behavior)

  - **FR-04-1**: **被动响应**: 标准 Modbus 读写响应。
  - **FR-04-2**: **主动变化**: 支持模拟电池电压随 SoC 变化的非线性曲线（可选），或模拟 PCS 启动时的软启动斜坡（Ramp up）。

  ### FR-05: 时间基准 (Time Basis)

  - **FR-05-1**: **MVP 实时仿真**: 物理引擎与脚本执行以真实时间为基准，默认 `time_scale = 1`。
  - **FR-05-2**: **可选加速预留**: 预留 `time_scale` 配置项（暂不实现），用于后续支持仿真时间加速或减速。

  ### FR-06: 性能与规模 (Scale & Performance)

  - **FR-06-1**: **MVP 规模目标**: 单端口 32 个 Slave、并发连接 100、稳定 QPS 200、P95 响应时间 < 50ms。
  - **FR-06-2**: **可扩展上限**: 通过配置项调整上限，架构需避免硬编码。

  ------

  ## 3.1 协议与数据模型补充 (Protocol & Data Model)

  ### PR-01: Modbus 功能码支持 (Function Codes)

  - **PR-01-1**: 支持 `0x03` 读保持寄存器、`0x04` 读输入寄存器、`0x06` 写单寄存器、`0x10` 写多寄存器。
  - **PR-01-2**: 其它功能码默认返回异常码 `0x01`（非法功能）。

  ### PR-02: 异常码映射 (Exception Mapping)

  - **PR-02-1**: 非法地址 -> `0x02`。
  - **PR-02-2**: 非法数值范围 -> `0x03`。
  - **PR-02-3**: 设备忙或内部异常 -> `0x06`。

  ### PR-03: 字节序与拼接 (Endianness)

  - **PR-03-1**: 16 位寄存器默认大端序。
  - **PR-03-2**: 32 位拼接默认高字在前（可在配置中覆盖）。
  - **PR-03-3**: 浮点类型采用 IEEE754，默认高字在前（可配置）。

  ### PR-04: 寄存器模型 (Register Model)

  - **PR-04-1**: 支持 `holding` / `input` / `coil` / `discrete` 四类寄存器。
  - **PR-04-2**: 每个寄存器需标注 `读写权限`、`单位`、`缩放因子`、`默认值`、`最小/最大值`。
  - **PR-04-3**: 对只读寄存器的写入请求返回异常码 `0x03`。

  ------

  ## 4. 数据模型与配置规范 (Configuration Specification)

  配置文件需支持 **设备组 (Device Group)** 的概念，以模拟总线。

  ### 4.0 配置约束 (Schema & Defaults)

  - **CFG-01**: `listeners.port`、`devices.name`、`devices.slave_id`、`profiles.*.registers` 为必填项。
  - **CFG-02**: `slave_id` 范围 1~247，超出范围视为非法配置。
  - **CFG-03**: 允许 `profiles` 复用；同名寄存器地址冲突时报错。
  - **CFG-04**: 配置热加载为 **可选增强**，MVP 默认不支持热加载。

  ### 4.1 典型配置示例：EMS 储能系统

  YAML

  

  ```
  # 定义总线监听器
  listeners:
    - port: 5020
      type: "tcp"
      description: "模拟 RS485 储能子系统 #1"
      devices: ["PCS_Unit_1", "BMS_Unit_1"] # 此端口下挂载的设备名
  
  # 定义具体的设备模型
  devices:
    - name: "PCS_Unit_1"
      slave_id: 1            # 在 5020 端口下的 ID 1
      profile: "pcs_100kw_v1" # 引用模板
  
    - name: "BMS_Unit_1"
      slave_id: 2            # 在 5020 端口下的 ID 2
      profile: "lfp_battery_v1"
  
  # 定义模板 (Profile)
  profiles:
    pcs_100kw_v1:
      registers:
        - address: 100
          name: "Active_Power_Set"
          type: "int16"
          min: -100
          max: 100
          description: "有功功率设定 kW (正=充, 负=放)"
  
    lfp_battery_v1:
      registers:
        - address: 200
          name: "SOC"
          type: "uint16"
          min: 0
          max: 1000
          description: "荷电状态 (0.0% - 100.0%)"
        
        - address: 201
          name: "Total_Voltage"
          type: "uint16"
          default: 7000 # 700.0V
  
  # 定义全局物理仿真脚本
  simulation:
    - name: "Battery_Charging_Logic"
      interval: 1.0 # 1秒执行一次
      # time_scale: 1.0 # 预留：仿真时间倍率（MVP 不启用）
      script: |
        # 1. 获取 PCS 的功率 (设备名, 地址)
        power = get_value("PCS_Unit_1", 100) 
        
        # 2. 获取 BMS 当前 SoC
        current_soc = get_value("BMS_Unit_1", 200) / 10.0
        
        # 3. 物理常数
        capacity_kwh = 200.0
        dt_hours = 1.0 / 3600.0
        
        # 4. 计算 SoC 增量
        # 功率(kW) * 时间(h) / 容量(kWh) * 100%
        delta_soc = (power * dt_hours / capacity_kwh) * 100.0
        
        # 5. 更新 SoC，并写入 BMS
        new_soc = current_soc + delta_soc
        
        # 边界保护
        if new_soc > 100.0: new_soc = 100.0
        if new_soc < 0.0: new_soc = 0.0
        
        set_value("BMS_Unit_1", 200, int(new_soc * 10))
        
        # 6. 简单的电压跟随逻辑 (SoC 越高电压越高)
        new_voltage = 600.0 + (new_soc * 1.5) 
        set_value("BMS_Unit_1", 201, int(new_voltage * 10))
  ```

  ## 5. 详细用例场景 (Use Case Scenario)

  ### UC-01: EMS 下发充电指令测试

  - **前置条件**: 模拟器启动，监听 TCP 5020。EMS 系统已连接 5020 端口。
  - **步骤**:
    1. EMS 向 **Slave ID 1** (PCS) 的寄存器 **100** 写入数值 `80` (代表 80kW)。
    2. 模拟器接收写入请求，校验数值在 -100 到 100 之间，写入成功。
    3. 模拟器后台线程检测到 PCS 功率为 80kW。
    4. 后台线程每秒计算电量积分，逐步增加 **Slave ID 2** (BMS) 的寄存器 **200** (SoC) 的数值。
    5. EMS 定时轮询 **Slave ID 2** 的寄存器 **200**。
  - **预期结果**: EMS 能够读到 BMS 的 SoC 在缓慢上升（如 50.0% -> 50.1% -> 50.2%），验证了 EMS 的充电策略有效性。

  ### UC-02: 电池满充保护测试

  - **步骤**:
    1. 手动或自动将 BMS SoC 模拟至 99%。
    2. EMS 继续下发充电指令。
    3. 仿真脚本检测到 SoC >= 100%。
    4. 仿真脚本强制将 PCS (Slave 1) 的功率反馈或实际输出置为 0，并设置 BMS (Slave 2) 的“充电禁止”告警位。
  - **预期结果**: EMS 读取到告警位，并停止下发功率指令。

  ## 6. 开发优先级 (Development Roadmap)

  1. **P0 (核心架构)**: 实现基于 `pymodbus` 的 `ServerContext`，支持单端口映射多个 `SlaveContext`。
  2. **P1 (物理引擎)**: 实现后台 Loop 线程，并解析 YAML 中的 `simulation` 脚本段落（使用 Python `exec` 或简单解析器）。
  3. **P2 (配置解析)**: 完成 YAML 文件到内存对象的映射。

  ------

  ## 7. 非功能需求 (Non-Functional Requirements)

  - **NFR-01 稳定性**: 连续运行 7x24 小时无崩溃。
  - **NFR-02 资源上限**: 单进程内存 < 512MB，CPU 峰值 < 70%（MVP 规模目标下）。
  - **NFR-03 可复现性**: 相同配置与输入下仿真结果可复现（允许随机种子配置）。
  - **NFR-04 可维护性**: 核心模块需有清晰边界与日志定位点。

  ------

  ## 8. 脚本执行与安全 (Scripting & Safety)

  - **SEC-01**: 脚本仅允许访问仿真接口（`get_value/set_value`），禁止文件与网络访问。
  - **SEC-02**: 单次脚本执行超时保护（默认 100ms），超时则跳过本轮并记录日志。
  - **SEC-03**: 脚本异常不影响主服务，需降级并持续运行。

  ------

  ## 9. 可观测性 (Observability)

  - **OBS-01**: 日志分级（INFO/WARN/ERROR），记录关键事件：写入失败、越界、脚本异常。
  - **OBS-02**: 运行指标（连接数、QPS、异常码计数、脚本耗时）可在日志中输出或导出。

  ------

  ## 10. 部署与运行 (Deployment)

  - **DEP-01**: 支持命令行参数指定配置路径与监听端口覆盖。
  - **DEP-02**: 前台与后台模式均可运行（MVP 仅需前台）。

  ------

  ## 11. 兼容性与依赖 (Compatibility)

  - **ENV-01**: Python 版本 **3.14**。
  - **ENV-02**: 依赖 `pymodbus`（版本范围在实现阶段明确）。
  - **ENV-03**: 兼容 Linux (CentOS/Ubuntu/Debian) 与 Windows。

  ## 12. 测试与验收 (Testing & Acceptance)

  - **TST-01**: 单端口多从站读写正确性。
  - **TST-02**: 范围限制与异常码返回正确。
  - **TST-03**: 物理引擎积分逻辑正确（SoC 单调变化）。
  - **TST-04**: 性能基准满足 FR-06 规模目标。

  ## 13. 术语表 (Glossary)

  - **EMS**: 能量管理系统。
  - **PCS**: 储能变流器。
  - **BMS**: 电池管理系统。
